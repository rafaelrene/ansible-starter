# Code Review

First, verify the current directory is inside a Git repository. If not, tell the user this skill can only be run inside a Git repository and stop.

If there are no local changes that have not yet been pushed to the remote (no staged, unstaged, or untracked files), tell the user there is nothing to review and stop.

Otherwise, review every file that has unpushed changes. Produce a single self-contained HTML report saved to `code-review.html` in the project root. The HTML must be a complete, standalone document with all CSS inlined — no external dependencies.

## Scope

The review covers all local changes that have not yet been pushed to the remote repository. This includes staged and unstaged modifications, as well as newly added (untracked) files.

What to skip:
- **Binary files** (images, PDFs, archives, etc.) — note their presence but do not review content.
- **Auto-generated files and build artifacts** — lock files (`package-lock.json`, `yarn.lock`, `Cargo.lock`, etc.), skill output (`code-review.html`), build output directories (`dist/`, `build/`, `target/`, etc.), compiler output, and any file marked as generated.

When an auto-generated file changes (e.g. a lock file), note the change briefly but do not perform a detailed review unless the diff suggests a specific concern (e.g. a removed dependency with no replacement).

## Review Sections

For each file, go through all three sections below. Be specific with line numbers, explain the issue, and provide a concrete fix or alternative. If a section has no findings, it should still appear with the text "None found."

### 1. Concerns (Box)
Code that needs attention:
- Potential bugs, logic errors, race conditions
- Over-engineering or excessive abstraction that "boxes in" future changes
- Violations of language idioms or project conventions
- Missing error handling, missing edge cases, resource leaks
- Security or privacy issues
- Performance problems (N+1 queries, unnecessary re-renders, etc.)
- Dead code, commented-out code, debug leftovers
- Type safety issues, improper null/undefined handling

Each finding must include: what the problem is, why it matters, and how to fix it. Include a before/after diff when the fix is non-trivial.

### 2. Simplification Opportunities
Code that could be cleaner:
- Duplicated logic that should be extracted or composed
- Overly complex conditionals, nested loops, callback chains
- Verbose patterns with cleaner modern alternatives
- Redundant abstractions (wrapper functions, unnecessary indirection)
- Unnecessarily mutable code where immutable would be clearer

Show the simpler version. Include a side-by-side or inline diff.

### 3. Warnings (Does This Make Sense?)
Suspicious-looking code flagged for author review:
- Assigned but unused variables or dead code branches
- Functions defined but never called (unless public API)
- Comparisons always true/false, unreachable branches
- Arguments accepted but never used
- `catch` blocks that swallow errors silently
- Async operations without `await` (or `await` on non-promises)
- Inverted boolean/numeric logic
- Placeholder or debug values left in
- Any code that looks unintentional

These are warnings, not definite bugs. Flag for the author to confirm intent.

## HTML Report Structure

Create a clean, readable report following the structure demonstrated in the reference example. See [`example.html`](example.html) in this skill directory for a complete, working implementation.

The report must include these sections in order:

1. **Header** — project name, date, reviewer info.
2. **Summary bar** — 4 cards showing total concerns, simplifications, warnings, and files reviewed. Use distinct colors per category (e.g. red/rose for concerns, amber/yellow for simplifications, orange for warnings).
3. **Per-file cards** — one card per file. File path as heading, then three sections (Concerns, Simplifications, Warnings) as distinct visual blocks.
   - Prefix each finding with an icon or colored badge.
   - Always show exact line numbers.
   - Use side-by-side `<table>` or inline diff with `-` red / `+` green markers for before/after. Wrap code in `<code>` or `<pre>` with monospace font.
4. **Footer** — timestamp, reviewer note, "generated by pi/code-review".

### CSS style guidelines

- Dark or light theme is fine — pick one that makes diffs and severity colors pop
- Use `system-ui` font stack
- Keep it clean and readable, not over-designed
- Responsive: works on mobile and desktop
- Print-friendly (media query for print)

### Output

Write the HTML file to `code-review.html` in the project root. After writing, run the open command:

- macOS: `open code-review.html`
- Linux: `xdg-open code-review.html`
- Windows: `start code-review.html`

### Replacing TODOs

The template file [`example.html`](example.html) contains `<!-- TODO: ... -->` markers for every piece of content that must be replaced with actual review data. Before writing the report, copy the template and replace all TODO markers with real content. Use the commented-out example file card in the template as a reference for the expected format.

When generating the report:

1. Copy the `<div class="file-card">...</div>` block from the template for each file reviewed.
2. Remove the `<!-- ... -->` comment wrapper around the block.
3. Replace every `<!-- TODO: ... -->` with actual findings, line numbers, code diffs, and counts.
4. Files with no findings in a section should use `<div class="none-found">None found.</div>` instead of a `<div class="finding">` block.
5. Update the summary card counts and header metadata to match.

## Principles

1. **Be specific.** Exact line numbers, exact suggestions.
2. **Prioritize.** Lead with the most important finding in each section.
3. **Be constructive.** Every concern must include a fix or alternative.
4. **Don't nitpick formatting.** Skip: indentation, whitespace, bracket placement, line length, comment style. Do flag: naming conventions that hurt readability, inconsistent patterns across files, violations of the project's documented style.
5. **Say "None found"** when a section is empty — visible in the HTML, not omitted.
